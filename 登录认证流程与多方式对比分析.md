## 整体流程总览

```
前端登录页面
      │
      ▼
authenticationService.login() / superAdminLogin()
      │
      ▼  POST /authenticate[/{organizationId}] 或 /authenticate/super-admin
AuthController.login() / superAdminLogin() / organizationLogin()
      │
      ▼
AuthService.login()
      │
      ├─► 是 invite redirect? ──► 特殊路径：查询 invited 状态用户、不校验 form SSO
      │
      ├─► 是 organizationId 指定? ──► 指定组织登录：校验 form SSO 启用状态
      │
      └─► 全局登录 ──► 查找支持 form 登录的组织 → 默认组织 → 首个可用 → 创建个人工作区
      │
      ▼
AuthUtilService.validateLoginUser()  ←──── 密码校验 + 重试计数
      │
      ▼
SessionUtilService.generateLoginResultPayload()
      │
      ├─► 创建/复用 UserSessions 记录
      ├─► 构建 JWTPayload
      ├─► 签发 JWT (jwtService.sign)
      └─► 设置 tj_auth_token cookie
      │
      ▼
返回登录结果（组织信息、权限、用户信息等）
```

## 后端认证链路

### 1. AuthController（路由层）

**文件**：[controller.ts](file:///Users/zhangjing/Desktop/so-coders/0609-under/ToolJet/server/src/modules/auth/controller.ts)

三个登录路由：

| 路由 | 方法 | 对应 Controller 方法 | Feature Key |
|------|------|---------------------|-------------|
| `POST /authenticate` | 全局登录 | `login()` | `FEATURE_KEY.LOGIN` |
| `POST /authenticate/super-admin` | 超管登录 | `superAdminLogin()` | `FEATURE_KEY.SUPER_ADMIN_LOGIN` |
| `POST /authenticate/:organizationId` | 指定组织登录 | `organizationLogin()` | `FEATURE_KEY.ORGANIZATION_LOGIN` |

所有接口均使用 `@Res({ passthrough: true })` 注入 Response 对象，用于设置 cookie。

### 2. AuthService（业务逻辑层）

**文件**：[service.ts](file:///Users/zhangjing/Desktop/so-coders/0609-under/ToolJet/server/src/modules/auth/service.ts)

核心方法：`login(response, appAuthDto, organizationId?, loggedInUser?)`

#### 2.1 Invite Redirect 判断

```typescript
const isInviteRedirect =
  redirectTo?.startsWith('/organization-invitations/') || redirectTo?.startsWith('/invitations/');
```

**是 invite redirect 时**：
- 查询用户时包含 `WORKSPACE_USER_STATUS.INVITED` 状态
- 直接使用 bcrypt 比较密码，**不经过 `validateLoginUser`**
- `organizationId` 被重置为 `undefined`（走全局登录分支确定目标组织）
- 保存 `invitingOrganizationId` 供后续使用

**非 invite redirect 时**：
- 调用 `authUtilService.validateLoginUser()` 进行完整校验

#### 2.2 全局登录分支（无 organizationId）

执行步骤：

1. **查找支持 form 登录的组织列表**
   - 调用 `organizationRepository.findOrganizationWithLoginSupport(user, 'form')`
   - 非超管用户：只返回 `ssoConfigs.sso='form' 且 enabled=true` 的组织

2. **确定目标组织**（优先级从高到低）：
   - 用户 `defaultOrganizationId` 对应的组织（且未归档）
   - 列表中第一个活跃（非 archived）组织
   - 若 `allowPersonalWorkspace` 且非 invite redirect → **自动创建个人工作区**
   - 都没有 → 抛出 `UnauthorizedException`

3. **密码域名校验**（EE 版本）
   - 调用 `validatePasswordDomain(email, orgAllowed, orgRestricted, instanceSettings)`
   - 组织级配置优先于实例级配置
   - 超管用户跳过此校验

#### 2.3 指定组织登录分支（有 organizationId）

执行步骤：

1. 直接设置 `user.organizationId = organizationId`
2. 校验该组织的 **form SSO 配置是否启用**
   ```typescript
   const formConfigs: SSOConfigs = organization?.ssoConfigs?.find((sso) => sso.sso === 'form');
   if (!formConfigs?.enabled) {
     throw new UnauthorizedException('Password login is disabled for the organization');
   }
   ```
3. 密码域名校验（同全局登录）

#### 2.4 登录后处理

- 若默认组织 ID 变化 → 更新 `defaultOrganizationId`
- 重置 `passwordRetryCount: 0`
- 清空 `forgotPasswordToken`
- 写入审计日志（非 invite redirect 时）
- 调用 `sessionUtilService.generateLoginResultPayload()` 生成最终结果

#### 2.5 superAdminLogin 方法

```typescript
async superAdminLogin(response: Response, appAuthDto: AppAuthenticationDto) {
  const { email } = appAuthDto;
  const user = await this.userRepository.findByEmail(email);

  if (!user || !isSuperAdmin(user)) {
    throw new UnauthorizedException('Only super admin can login through this url');
  }

  return this.login(response, appAuthDto); // 复用全局登录逻辑
}
```

关键点：先校验用户是否为超管 → 然后走完全相同的 `login()` 流程。

### 3. AuthUtilService（认证工具服务）

**文件**：[util.service.ts](file:///Users/zhangjing/Desktop/so-coders/0609-under/ToolJet/server/src/modules/auth/util.service.ts)

核心方法：`validateLoginUser(email, password, organizationId?)`

#### 密码重试限制逻辑

```typescript
const passwordRetryConfig = this.configService.get<string>('PASSWORD_RETRY_LIMIT');
const passwordRetryAllowed = passwordRetryConfig ? parseInt(passwordRetryConfig) : 5;

if (
  this.configService.get<string>('DISABLE_PASSWORD_RETRY_LIMIT') !== 'true' &&
  user.passwordRetryCount >= passwordRetryAllowed
) {
  throw new UnauthorizedException(
    'Maximum password retry limit reached, please reset your password using forgot password option'
  );
}

if (!(await bcrypt.compare(password, user.password))) {
  await this.userRepository.update(user.id, { passwordRetryCount: user.passwordRetryCount + 1 });
  throw new UnauthorizedException('Invalid credentials');
}
```

规则总结：

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `PASSWORD_RETRY_LIMIT` | 5 次 | 最大重试次数 |
| `DISABLE_PASSWORD_RETRY_LIMIT` | false | 设置为 'true' 可禁用重试限制 |
| 重试计数时机 | 密码错误时 +1 | 每次错误密码尝试递增 |
| 重置时机 | 登录成功时归零 | 成功登录后 `passwordRetryCount = 0` |
| 达到上限后 | 需通过"忘记密码"重置 | 不提供自动解锁 |

### 4. SessionUtilService（会话工具服务）

**文件**：[util.service.ts](file:///Users/zhangjing/Desktop/so-coders/0609-under/ToolJet/server/src/modules/session/util.service.ts)

核心方法：`generateLoginResultPayload()`

#### 4.1 Session 创建逻辑

```typescript
let sessionId = isPatLogin ? user.sessionId : loggedInUser?.sessionId;
if (loggedInUser?.id !== user.id && !isPatLogin) {
  // 创建新的 UserSessions 记录
  const session: UserSessions = await this.createSession(user.id, deviceInfo, manager);
  sessionId = session.id;
}
```

- **新用户登录**（首次登录或切换账号）：创建新 session
- **已登录用户切换组织**：复用现有 session（`loggedInUser.id === user.id`）
- **PAT 登录**：走独立分支，不设置 cookie

#### 4.2 createSession 方法

```typescript
async createSession(userId: string, device: string, manager?: EntityManager): Promise<UserSessions> {
  const session = await manager.save(manager.create(UserSessions, {
    userId,
    device,
    createdAt: new Date(),
    expiry: this.getSessionExpiry(),
    lastLoggedIn: new Date(),
  }));
  // 递增 active sessions 指标
  incrementActiveSessions({ userId, sessionType: 'user' });
  return session;
}
```

Session 过期时间：
- 默认：10 天（14400 分钟）
- 可通过 `USER_SESSION_EXPIRY` 环境变量配置（单位：分钟）

## JWT Payload 与 Cookie 选项

### 1. JWTPayload 结构

**接口定义**：[IService.ts](file:///Users/zhangjing/Desktop/so-coders/0609-under/ToolJet/server/src/modules/session/interfaces/IService.ts#L27-L36)

```typescript
export interface JWTPayload {
  sessionId: string;           // UserSessions 记录 ID
  username: string;            // 用户 ID (user.id)
  sub: string;                 // 用户邮箱
  organizationIds: string[];   // 用户可访问的组织 ID 列表
  isSSOLogin: boolean;         // 是否为 SSO 登录
  isPasswordLogin: boolean;    // 是否为密码登录
  invitedOrganizationId?: string; // 受邀组织 ID（可选）
  isPatLogin?: boolean;        // 是否为 PAT 登录（可选）
}
```

**实际构建代码**（SessionUtilService.generateLoginResultPayload）：

```typescript
const JWTPayload: JWTPayload = {
  sessionId: sessionId,
  username: user.id,
  sub: user.email,
  organizationIds: [...organizationIds],
  isSSOLogin: isPatLogin ? false : loggedInUser?.isSSOLogin || isInstanceSSO,
  isPasswordLogin: isPatLogin ? false : loggedInUser?.isPasswordLogin || isPasswordLogin,
  ...(invitedOrganizationId ? { invitedOrganizationId } : {}),
  ...(isPatLogin ? { isPATLogin: true } : {}),
  ...(isPatLogin && extraData?.token ? { token: extraData?.token } : {}),
  ...(isPatLogin && extraData?.appId ? { appId: extraData?.appId, scope: 'App' } : {}),
  ...(extraData?.tj_api_source ? { tj_api_source: extraData.tj_api_source } : {}),
};
```

注意：PAT 登录时 `isSSOLogin` 和 `isPasswordLogin` 都设为 `false`。

### 2. Cookie 选项

**tj_auth_token Cookie 配置**（SessionUtilService.generateLoginResultPayload）：

```typescript
const cookieOptions: CookieOptions = {
  secure: isHttpsEnabled(),       // 取决于 TOOLJET_HOST 是否为 https 开头
  httpOnly: true,                 // 禁止 JS 访问
  sameSite: 'strict',             // 严格同源策略
  maxAge: 2 * 365 * 24 * 60 * 60 * 1000, // 最大 2 年（浏览器端）
};
```

#### 条件覆盖的配置

| 条件 | sameSite | secure | 说明 |
|------|----------|--------|------|
| 默认 | `strict` | 取决于 HTTPS | 最严格的安全配置 |
| `ENABLE_PRIVATE_APP_EMBED === 'true'` | `none` | `true` | 支持私有应用嵌入（iframe 跨站） |
| `ENABLE_CUSTOM_DOMAINS === 'true'` 且 HTTPS | `none` | `true` | 支持自定义域名跨站 |

#### isHttpsEnabled 判定

#### applyCustomDomainCookieOptions

**文件**：[utils.helper.ts](file:///Users/zhangjing/Desktop/so-coders/0609-under/ToolJet/server/src/helpers/utils.helper.ts#L640-L648)

```typescript
export const applyCustomDomainCookieOptions = (
  cookieOptions: { sameSite?: string | boolean; secure?: boolean },
  configService: { get<T>(key: string): T }
) => {
  if (configService.get<string>('ENABLE_CUSTOM_DOMAINS') === 'true' && isHttpsEnabled()) {
    cookieOptions.sameSite = 'none';
    cookieOptions.secure = true;
  }
};
```

#### 登出时的 cookie 清除

```typescript
getClearCookieOptions(): CookieOptions {
  const cookieOptions: CookieOptions = {
    httpOnly: true,
    secure: isHttpsEnabled(),
    sameSite: 'strict',
  };
  // 同样应用 ENABLE_PRIVATE_APP_EMBED 和自定义域名逻辑
  if (this.configService.get<string>('ENABLE_PRIVATE_APP_EMBED') === 'true') {
    cookieOptions.sameSite = 'none';
    cookieOptions.secure = true;
  }
  applyCustomDomainCookieOptions(cookieOptions, this.configService);
  return cookieOptions;
}
```

## 多种登录方式深度对比

### 对比总表

| 维度 | 全局登录 | 指定 organizationId 登录 | Invite Redirect | Super Admin 登录 |
|------|----------|------------------------|-----------------|-----------------|
| **API 端点** | `POST /authenticate` | `POST /authenticate/:organizationId` | `POST /authenticate` (带特殊 redirectTo) | `POST /authenticate/super-admin` |
| **前端入口** | `/login` | `/login/:organizationId` | 邀请邮件链接跳转 | `/super-admin/login` |
| **用户校验方法** | `validateLoginUser` | `validateLoginUser` | 直接 bcrypt 比较 | 先校验超管身份 → 走全局登录 |
| **可接受的用户状态** | ACTIVE, ARCHIVED | ACTIVE, ARCHIVED | INVITED (有密码) | ACTIVE |
| **组织选择逻辑** | 找 form 登录可用的组织 | 直接使用指定组织 | 走全局逻辑但记录 invitingOrgId | 走全局逻辑 |
| **Form SSO 检查** | 是（筛选组织时） | 是（显式检查） | 否 | 是（筛选组织时，超管跳过） |
| **密码重试限制** | 有 | 有 | 无（直接比较） | 有 |
| **密码域名校验** | 有（组织级） | 有（组织级） | 有（全局分支中） | 跳过（超管） |
| **个人工作区创建** | 条件触发 | 不触发 | 不触发 | 条件触发 |
| **审计日志** | 有 | 有 | 无 | 有 |
| **JWT isPasswordLogin** | true | true | true | true |
| **JWT isSSOLogin** | false | false | false | false |

### 1. 全局登录（Global Login）

**触发条件**：无 `organizationId` 参数

**组织选择算法**：

```
支持 form 登录的组织列表
    │
    ├─► 有 defaultOrganizationId 且未归档 → 使用它
    │
    ├─► 否则，取第一个活跃（非 archived）组织
    │
    ├─► 否则，allowPersonalWorkspace 且非 invite redirect → 创建个人工作区
    │
    └─► 否则 → 报错 "User is not assigned to any workspaces"
```

**Form SSO 配置作用点**：
- 在 `findOrganizationWithLoginSupport(user, 'form')` 中筛选
- 非超管用户：仅返回 `sso='form' 且 enabled=true` 的组织
- 超管用户：不受 SSO 配置限制，返回所有关联组织

### 2. 指定 organizationId 登录

**触发条件**：URL 或请求带 `organizationId`

**关键差异点**：

1. **显式 Form SSO 检查**
   - 必须检查目标组织的 `form` SSO 配置是否启用
   - 未启用直接报错："Password login is disabled for the organization"

2. **组织固定**
   - 不做组织选择，直接使用指定的 organizationId
   - 不触发个人工作区创建

3. **用户状态校验**
   - `validateLoginUser` 中会检查该组织下用户是否被归档
   - 归档用户报错："You have been archived from this workspace..."

### 3. Invite Redirect 登录

**触发条件**：`redirectTo` 以 `/organization-invitations/` 或 `/invitations/` 开头

**特殊处理逻辑**：

| 环节 | 普通登录 | Invite Redirect |
|------|----------|-----------------|
| **用户查询状态** | ACTIVE, ARCHIVED | 增加 INVITED |
| **密码校验方式** | validateLoginUser（带重试） | 直接 bcrypt.compare（无重试计数） |
| **密码重试限制** | 有 | 无（直接比较） |
| **organizationId 处理** | 无则全局查找 | 重置为 undefined，走全局逻辑 |
| **审计日志** | 有 | 无 |
| **无可用组织时** | 报错 | 不报错，保留 invitingOrgId |

**设计意图**：受邀用户在接受邀请前需要先登录，但此时用户在受邀组织中可能还是 `INVITED` 状态，需要特殊放行。

### 4. Super Admin 登录

**触发条件**：访问 `/authenticate/super-admin` 端点

**关键差异点**：

1. **前置校验**：必须先确认用户是超管，否则直接报错
2. **复用逻辑**：校验通过后完全走全局登录流程
3. **超管特权**：
   - 跳过 form SSO 筛选（能看到所有关联组织）
   - 跳过密码域名校验
   - 切换工作区时绕过登录方式检查

### 5. 个人工作区自动创建

**触发条件**（同时满足）：
- 全局登录路径
- 没有找到任何可用的 form 登录组织
- `allowPersonalWorkspace === true`
- 非 invite redirect

**allowPersonalWorkspace 判定**：
```typescript

  isSuperAdmin(user) ||
  (await this.instanceSettingsUtilService.getSettings(INSTANCE_USER_SETTINGS.ALLOW_PERSONAL_WORKSPACE)) === 'true';
```

即：**超管始终允许**，普通用户取决于实例配置 `ALLOW_PERSONAL_WORKSPACE`。

### 6. Form SSO 配置详解

**配置存储**：`sso_configs` 表，`sso='form'`

**组织创建时默认值**：
```typescript
ssoConfigs: [{
  sso: SSOType.FORM,
  enabled: true,
  configScope: ConfigScope.ORGANIZATION,
}]
```

即：**新建组织默认启用密码登录**。

**作用位置**：

| 场景 | 检查方式 |
|------|----------|
| 全局登录选组织 | `findOrganizationWithLoginSupport(user, 'form')` 中作为筛选条件 |
| 指定组织登录 | 显式检查 `formConfigs?.enabled` |
| 切换工作区 | 检查 form 登录启用状态（超管跳过） |
| SSO 登录选组织 | 使用 `loginType='sso'` 检查 `inheritSSO` |

### 7. 密码域名限制（validatePasswordDomain）

**文件**：[utils.helper.ts](file:///Users/zhangjing/Desktop/so-coders/0609-under/ToolJet/server/src/helpers/utils.helper.ts#L797-L825)

**层级关系**：
- 组织级配置（`passwordAllowedDomains`、`passwordRestrictedDomains`）**优先**于实例级配置
- 组织级有配置 → 使用组织级
- 组织级无配置 → 回退到实例级

**CE 版本**：直接返回 `true`（不启用此功能）

**校验规则**（由 `isValidPasswordDomain` 实现）：
1. 如有允许域名列表 → 仅允许列表中的域名
2. 如有禁止域名列表 → 排除列表中的域名
3. 都没有 → 全部允许

**文件**：[session/util.service.ts](file:///Users/zhangjing/Desktop/so-coders/0609-under/ToolJet/server/src/modules/session/util.service.ts#L513-L570)

async validateUserSession(userId, sessionId, organizationId?)
