## 4. StripeWebhookGuard 审计

### 4.1 代码位置

[stripe-webhook.guard.ts](file:///Users/zhangjing/Desktop/so-coders/0609-under/ToolJet/server/src/modules/organization-payments/guards/stripe-webhook.guard.ts)

### 4.2 核心实现

```typescript
@Injectable()
export class StripeWebhookGuard implements CanActivate {
  async validateStripeEvent(payload): Promise<boolean> {
    const endpointSecret = this.configService.get<string>('STRIPE_WEBHOOK_ENDPOINT_SECRET');
    const stripeAPIKey = this.configService.get<string>('STRIPE_API_KEY');
    const stripe = new Stripe(stripeAPIKey);
    const header = stripe.webhooks.generateTestHeaderString({
      payload: payload,
      secret: endpointSecret,
    });
    if (endpointSecret) {
      try {
        stripe.webhooks.constructEvent(payload, header, endpointSecret);
        return true;
      } catch (err) {
        console.log(`⚠️  Webhook signature verification failed.`, err.message);
        return false;
      }
    }
    return false;
  }

  async canActivate(context: ExecutionContext): Promise<boolean> {
    const request = context.switchToHttp().getRequest();
    const payload = request.rawBody || JSON.stringify(request.body);
    const isStripeEvent = await this.validateStripeEvent(payload);
    return isStripeEvent;
  }
}
```

### 4.3 严重安全问题：使用 generateTestHeaderString

**问题核心**：该 guard 使用 `stripe.webhooks.generateTestHeaderString()` 来生成签名 header，然后用 `stripe.webhooks.constructEvent()` 验证。

这是完全无效的验证方式，原因如下：

1. **自签名验证**：`generateTestHeaderString` 是 Stripe SDK 提供的**测试工具函数**，用于在测试环境中生成有效的签名头部。它使用相同的 endpoint secret 对 payload 进行签名。
2. **没有使用真实的 Stripe 签名**：正确的做法应该是从 HTTP 请求头 `stripe-signature` 中获取 Stripe 官方生成的签名，而不是自己生成。
3. **任何 payload 都能通过验证**：因为签名是自己生成的，任何传入的 payload 都会被"验证"通过，完全失去了 webhook 签名验证的意义。

**正确实现应该是**：

```typescript
const sig = request.headers['stripe-signature'];
const event = stripe.webhooks.constructEvent(payload, sig, endpointSecret);
```

### 4.4 rawBody 处理的潜在问题

1. **fallback 机制**：`payload = request.rawBody || JSON.stringify(request.body)` 中，如果 `rawBody` 不存在会 fallback 到 `JSON.stringify(request.body)`。
2. **JSON.stringify 可能改变原始字节**：Stripe 的签名验证是基于原始请求体的精确字节的。经过 JSON 解析再序列化后：
   - 键的顺序可能改变
   - 空白字符可能改变
   - Unicode 转义可能不同
   这些都会导致签名验证失败。
3. **rawBody 配置**：在 [bootstrap.helper.ts](file:///Users/zhangjing/Desktop/so-coders/0609-under/ToolJet/server/src/helpers/bootstrap.helper.ts#L38-L42) 的 `rawBodyBuffer` 函数中，通过 express json 中间件的 `verify` 回调来设置 `req.rawBody`。这个配置在 [main.ts](file:///Users/zhangjing/Desktop/so-coders/0609-under/ToolJet/server/src/main.ts#L210) 中启用。

### 4.5 Guard 使用情况

**注意**：在 CE 版本的代码库中搜索，`StripeWebhookGuard` 只被定义但**没有被实际使用**（没有 `@UseGuards(StripeWebhookGuard)` 装饰器）。这可能意味着：
- EE 版本的 controller 中会使用该 guard
- 或者该 guard 还未被集成

## 5. 组织访问 Guard 审计

### 5.1 OrganizationLicenseAccessGuard

[organizationLicenseAccess.guard.ts](file:///Users/zhangjing/Desktop/so-coders/0609-under/ToolJet/server/src/modules/organization-payments/guards/organizationLicenseAccess.guard.ts)

```typescript
@Injectable()
export class OrganizationLicenseAccessGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    const request = context.switchToHttp().getRequest();
    const user = request.user;
    const organizationId = request?.params?.organizationId;

    if (user.organizationId != organizationId) {
      throw new ForbiddenException("Access to the requested organization's data is forbidden");
    }
    return true;
  }
}
```

**功能**：验证当前用户的 `organizationId` 是否与请求路径参数中的 `organizationId` 匹配。

**潜在问题**：
1. 使用 `!=` 而非 `!==`，存在隐式类型转换风险
2. 没有检查 `user` 是否存在，若用户未认证可能导致运行时错误
3. 没有检查 `organizationId` 参数是否存在

### 7.4 为什么 CE 版本大多未实现

1. **开源核心 vs 企业版功能**：支付和订阅管理是企业版/云版的核心商业化功能，CE（社区版）不包含完整实现。
2. **接口契约先行**：CE 版本定义了完整的接口（Interface）和骨架实现，确保代码可以正常编译和运行。
3. **动态替换机制**：通过 `SubModule.getProviders()` 动态加载机制，EE 版本可以无缝替换 CE 版本的实现。
4. **保留扩展点**：CE 版本的模块结构为第三方开发者提供了扩展的可能性。

### 7.5 EE 实现连接方式

EE 版本需要在 `ee/organization-payments/` 目录下提供对应的实现文件：

- `ee/organization-payments/controller.ts` - EE 版控制器
- `ee/organization-payments/service.ts` - EE 版服务

当 `getTooljetEdition()` 返回 `ee` 或 `cloud` 时，`getImportPath()` 会返回 `{cwd}/dist/ee` 路径，`getProviders` 会从该路径下加载 controller 和 service。

## 9. 总结与风险评级

| 审计项 | 风险等级 | 说明 |
|--------|----------|------|
| StripeWebhookGuard 签名验证 | **严重** | 使用 generateTestHeaderString 自签名，完全无效 |
| rawBody fallback 机制 | 中 | JSON.stringify 可能导致签名验证不一致 |
| DTO 校验完整性 | 低 | 部分字段校验不严格 |
| OrganizationLicenseAccessGuard | 低 | 缺少空值检查，使用非严格相等 |
| CE 未实现方法 | 设计预期 | 社区版不包含支付功能实现 |
| 动态模块扩展机制 | 设计合理 | CE/EE 通过动态加载实现代码隔离 |

### 9.1 关键建议

1. **立即修复 StripeWebhookGuard**：
   - 从 `request.headers['stripe-signature']` 获取真实的签名
   - 移除 `generateTestHeaderString` 的使用
   - 确保使用原始 rawBody 进行验证

2. **确保 webhook 路由使用 rawBody**：
   - Stripe webhook 端点应该使用原始请求体，不要经过 JSON 解析
   - 或者确保 JSON 中间件正确保留了原始字节的 rawBody

3. **完善 DTO 校验**：
   - 统一字段类型（如 quantity 应为数字）
   - 添加必要的 @IsNotEmpty 校验
   - 避免使用 `any` 类型

4. **加强 OrganizationLicenseAccessGuard**：
   - 添加用户存在性检查
   - 使用严格相等比较
   - 添加参数存在性验证
